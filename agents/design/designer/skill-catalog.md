# Designer Skill Catalog

Read this file first when you are staffed for orchestrated work.
It lists only the role-local skills in this folder and keeps descriptions short so you can scan cheaply.
Open only the matching skill files under `skills/`, then end your closing handoff with `Read <skill-paths> skills for this task.`

## research/

- `research/plan` — Turn a research request into a defensible study plan with the right question, method, sample, and reporting path before any sessions run.
- `research/interview` — Prepare for, run, and debrief user interviews so the team learns how people actually work, think, and decide in context.
- `research/synthesize` — Turn research notes, transcripts, and observations into defensible themes, insights, and implications the design team can act on.
- `research/test` — Design and run usability tests that measure whether users can complete key tasks, where they struggle, and what the interface needs to change.

## ux/

- `ux/flow` — Design task flows that cover entry points, happy paths, alternates, errors, empty states, recovery, and completion for a product experience.
- `ux/frame` — Turn an ambiguous product request into a clear problem statement, scope boundary, success criteria, and design direction for the rest of the work.
- `ux/map` — Map each design decision to the source component, pattern, token, or exception it depends on so the handoff stays traceable.
- `ux/wire` — Translate task flows and information architecture into low-fidelity wireframes that define screen purpose, content hierarchy, and interaction scaffolding for user-centered evaluation.
- `ux/distill` — Remove unnecessary structural complexity so the experience becomes easier to understand, navigate, and complete.
- `ux/onboard` — Design onboarding, empty-state, and first-run experiences that get users to value quickly without overwhelming or patronizing them.
- `ux/validate` — Validate a UX direction with heuristics, scenarios, and usability checks before the design is handed off or implemented.
- `ux/delight` — Add well-judged moments of personality, reassurance, and reward that make the product feel more memorable without getting in the way of the task.
- `ux/specify` — Write a complete UI specification for screens and states, including structure, behavior, responsiveness, content, and interaction details.

## ui/

- `ui/stateful` — Specify the complete visual state model for a UI surface so every control, container, and feedback area has defined appearance, content, and transition behavior across all interactive and data-driven states.
- `ui/respond` — Define how the interface responds to user actions and system events with timely feedback, clear recovery paths, and behavior that feels immediate, understandable, and trustworthy.
- `ui/adapt` — Adapt an approved UI surface across breakpoints, devices, and contexts without losing hierarchy, usability, or design-system fidelity.
- `ui/arrange` — Improve layout, spacing, grouping, and visual rhythm so the surface feels intentional, legible, and structurally strong.
- `ui/bolder` — Increase visual impact and personality in safe or bland UI without sacrificing clarity, usability, or system coherence.
- `ui/colorize` — Introduce strategic color so the surface communicates more clearly and feels more expressive without drifting from brand, theme, or accessibility requirements.
- `ui/polish` — Run a final UI quality pass to correct spacing, alignment, states, typography, and other small details that separate acceptable work from ship-ready work.
- `ui/quieter` — Reduce visual intensity in an overstimulating interface while preserving hierarchy, clarity, and the product's intended character.
- `ui/typeset` — Improve typography so text feels deliberate, readable, and structurally clear across the full UI surface.
- `ui/frontend-design` — Set the overall visual direction for new UI work so the surface feels intentional, distinctive, brand-aligned, and ready for production handoff.

## content/

- `content/clarify` — Rewrite confusing labels, instructions, errors, and other interface copy so users understand what happened and what to do next.
- `content/harmonize` — Align terminology, tone, and copy patterns across a flow so the product sounds consistent from screen to screen and state to state.
- `content/name` — Create clear, scalable names for features, settings, concepts, and navigation labels that match user vocabulary and stay consistent over time.
- `content/orient` — Create onboarding and in-context guidance that helps users understand the value, next step, and safest path to progress without overexplaining.
- `content/recover` — Write error, empty, confirmation, and success messaging that helps users recover, proceed, or feel confident about what just happened.
- `content/structure` — Organize headings, labels, helper text, and content hierarchy so complex screens become easier to scan, understand, and act on.
- `content/scope` — Define the assistant's boundaries, supported intents, exclusions, and handoff rules for conversation design work.
- `content/script` — Draft and refine conversation scripts, dialogue trees, and turn-by-turn response sequences for AI-powered experiences.
- `content/tune` — Refine conversational voice, response calibration, and dialogue tone so the assistant sounds trustworthy, context-appropriate, and emotionally attuned across turns and states.

## motion/

- `motion/animate` — Add purposeful motion and micro-interactions that improve feedback, orientation, and perceived quality without harming performance or accessibility.
- `motion/reduce` — Design reduced-motion alternatives that preserve meaning while minimizing vestibular and attention load.
- `motion/sequence` — Plan ordered motion across multiple elements so timing, hierarchy, and attention flow feel deliberate.
- `motion/signal` — Design motion that signals status, urgency, attention, and meaning without overwhelming the interface.
- `motion/transition` — Design state-to-state motion that preserves continuity, clarifies cause and effect, and helps users understand what changed.

## accessibility/

- `accessibility/audit` — Audit designs, flows, and implementations for accessibility barriers, WCAG gaps, and assistive-technology regressions before they ship.
- `accessibility/annotate` — Add accessibility notes, constraints, and implementation guidance to designs, specs, and handoffs so teams can build them correctly.
- `accessibility/adapt` — Adapt designs, patterns, and interactions so they work for keyboard users, screen reader users, and people with low vision, motion sensitivity, or cognitive load constraints.
- `accessibility/verify` — Verify that accessibility fixes and specs actually work through manual checks, assistive technology testing, and repeatable acceptance criteria.

## architecture/

- `architecture/find` — Design the search and findability experience so users can locate content through search, browse, filters, and recovery paths with minimal friction.
- `architecture/label` — Design the exact names users see in navigation, facets, headings, and other structural surfaces so each label is distinct, predictable, and usable.
- `architecture/map` — Translate an information architecture problem into current-state and future-state structures with hierarchy, entry points, wayfinding, and findability constraints.
- `architecture/taxonomy` — Define the controlled vocabulary and classification scheme that keeps product content organized, searchable, and consistent across surfaces.

## localization/

- `localization/adapt` — Adjust product behavior, content structure, and visual treatment to fit cultural, legal, and regional expectations without losing product intent.
- `localization/expand` — Design interfaces to survive text expansion across locales without truncation, clipping, or fragile fixed-width assumptions.
- `localization/mirror` — Adapt layouts, navigation, and directional UI so RTL locales read and operate naturally instead of appearing flipped as an afterthought.
- `localization/pseudo` — Use pseudo-localization to expose layout breakage, clipping, and string-handling issues before real translations are available.

## service/

- `service/align` — Align service design decisions with operational reality, stakeholder needs, and service metrics so the proposed experience can actually be delivered and measured.
- `service/blueprint` — Create or revise end-to-end service blueprints that map customer actions, frontstage interactions, backstage processes, and support systems across the full journey.
- `service/handoff` — Design, clarify, and document transitions between channels, teams, systems, and ownership boundaries so context moves cleanly through the service.
- `service/surface` — Specify service touchpoints across digital, human, and physical surfaces so each interaction is owned, legible, and consistent within the broader service experience.

## Additional Skills

- `architecture/apply-patch` — Write the IA design package — task flows, site maps, navigation specs, and component mappings.
- `architecture/chrome-take-snapshot` — Capture the current product's navigation and structural DOM to understand the existing IA and identify patterns the new design must integrate with.
- `architecture/figma-get-design-context` — Retrieve existing navigation components, taxonomy structures, and IA patterns from the Figma design system before authoring new IA decisions.
- `content/apply-patch` — Write and update the conversation design artifacts — dialogue flows, prompt patterns, and fallback paths.
- `content/chrome-take-snapshot` — Capture the current DOM snapshot of a live product to inspect the conversation surface, UI states, and existing interaction patterns.
- `content/search-query` — Research conversation design patterns, dialogue frameworks, and AI interaction precedents before authoring flows.
- `localization/apply-patch` — Write the localization design package — locale adaptation specs, RTL layout guidelines, and component mapping for multi-locale support.
- `localization/figma-get-design-context` — Retrieve component specs, layout constraints, and token definitions to assess locale adaptation requirements before authoring localization design.
- `localization/figma-get-screenshot` — Capture locale variant frames from Figma to visually verify text expansion, RTL rendering, and cultural adaptation in context.
- `research/chrome-navigate-page` — Explore the current product and competitor experiences first-hand to generate observational evidence for research synthesis.
- `research/chrome-take-snapshot` — Capture DOM snapshots as durable research evidence — documenting current product states, competitor patterns, and observed friction points.
- `research/search-query` — Research user behavior patterns, existing studies, competitive UX analysis, and domain insights to generate evidence before design decisions are made.
- `service/apply-patch` — Write the service design package — blueprints, journey maps, cross-channel touchpoint specs, and backstage process documentation.
- `service/chrome-take-snapshot` — Capture live product touchpoints to document the current-state digital experience within the broader service context.
- `service/search-query` — Research service design patterns, cross-channel experience frameworks, and backstage process models before authoring blueprints.
- `ui/apply-patch` — Write the high-fidelity UI design package — visual surface specs, complete state coverage, and design-system-aligned component mappings.
- `ui/figma-get-design-context` — Retrieve the full design system context — components, tokens, variants, and spacing rules — before authoring any visual surface.
- `ui/image-query` — Search for visual design references and UI pattern examples to inform high-fidelity visual decisions and validate design choices.
- `ux/apply-patch` — Write the full UX/UI design package — task flows, UI spec, component mapping, and research rationale.
- `ux/figma-get-design-context` — Retrieve component specs, tokens, and design system patterns from Figma before authoring the UX/UI design package.
- `ux/image-query` — Search for and analyze visual design references, competitor UI patterns, and design system examples to inform and validate design decisions.

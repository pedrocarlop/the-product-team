# Map

## Purpose

Use this skill to turn a messy request into a clear structure: what the user is trying to do, what the system needs to support, and how the experience should be sequenced across screens or states.

## When to Use

- When the feature scope is unclear and needs a stronger task or flow definition
- When content, steps, or decisions need to be grouped into a usable information architecture
- When you need to identify the minimum set of screens, branches, and states for a design

## When Not to Use

- When the structure is already settled and the task is mainly layout or visual refinement
- When the problem is primarily wording, tone, or microcopy clarity
- When the work is about validating an existing flow rather than shaping it

## Required Inputs

- The product goal and the user problem it solves
- Any approved requirements, source notes, or existing flow context
- Known users, entry points, and exit conditions
- Constraints that affect structure, such as platform, device, or technical limits
- Screenshots, mocks, or examples if the current flow already exists

## Workflow

1. Identify the core user goal and the outcome the product must support.
2. List the actions, decisions, and system responses that are essential to completing that goal.
3. Group related content and steps into a logical task structure.
4. Determine the sequence, branches, and state changes the user can encounter.
5. Reduce unnecessary steps, duplicate decisions, and hidden dependencies.
6. Check the mapped structure against the requirement set and call out any gaps or assumptions.

## Design Principles / Evaluation Criteria

- Structure should match the user's mental model
- Every screen or step should earn its place
- Decisions should appear only when they matter
- Sequence should minimize backtracking and confusion
- The final map should be implementation-aware and easy to hand off

## Output Contract

- A task map or flow outline
- A screen or state inventory with key decisions and transitions
- Any unresolved questions or assumptions that affect the structure
- A short rationale for major grouping or sequencing choices

## Guardrails

- Do not design visuals before the flow is clear
- Do not hide important branches inside vague "other" buckets
- Do not invent product behavior that is not supported by the source material
- Do not overcomplicate the map with every possible edge case if it does not change the task

## Optional Tools / Resources

- Existing flow diagrams or journey maps
- Product requirements or discovery notes
- Screenshots of the current experience
- Research findings, support tickets, or analytics that show where the structure breaks down

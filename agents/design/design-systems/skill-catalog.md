# Design Systems Skill Catalog

Read this file first when you are staffed for orchestrated work.
It lists only the role-local skills in this folder and keeps descriptions short so you can scan cheaply.
Open only the matching skill files under `skills/`, then end your closing handoff with `Read <skill-paths> skills for this task.`

## system/

- `system/compose` — Build components and patterns from approved tokens and primitives so the design system has clear anatomy, variants, and composition rules.
- `system/document` — Write system documentation that explains when to use an asset, when not to use it, how it works, and what adopters need to know.
- `system/extract` — Identify reusable components, tokens, and patterns and promote them into the design system with clear ownership, structure, and migration intent.
- `system/govern` — Define contribution paths, ownership, lifecycle policy, and deprecation rules so the design system evolves without chaos.
- `system/normalize` — Bring a drifting feature back into alignment with the design system by replacing ad hoc decisions with approved components, tokens, and patterns.
- `system/tokenize` — Define, name, layer, and mode design tokens so the design system expresses values as a maintainable contract instead of scattered literals. Use when a feature has repeated hard-coded values, when light/dark modes need better structure, or when tokens are drift from design/code.

## technology/

- `technology/audit` — Run a technical UI quality audit across accessibility, responsiveness, performance, theming, and implementation anti-patterns, then report prioritized findings.
- `technology/bridge` — Translate design intent into implementation-ready guidance by mapping Figma specs, design tokens, component anatomy, states, and interaction details to the code surface that should carry them.
- `technology/harden` — Strengthen an interface against edge cases, failure states, localization pressure, and real-world data conditions so it remains reliable outside ideal demos.
- `technology/optimize` — Diagnose UI performance bottlenecks and define the highest-value changes needed to improve speed, smoothness, and perceived responsiveness.
- `technology/overdrive` — Explore technically ambitious UI directions and turn the best context-appropriate option into a feasible, progressive-enhancement-ready concept.
- `technology/prototype` — Build or specify high-fidelity prototypes that exercise real interactions, state transitions, and logic so design decisions can be validated before full implementation.
- `technology/verify` — Check that a design implementation matches the intended spec across layout, tokens, behavior, accessibility, and state coverage, then report the gaps that still need attention.

## direction/

- `direction/critique` — Review design work at the portfolio level to identify the highest-impact quality, strategy, and consistency issues and set the bar for what ships.
- `direction/mentor` — Coach designers through feedback, growth plans, and working sessions so they build judgment, confidence, and consistency.
- `direction/standardize` — Define and document repeatable design standards so quality, critique, and handoff stay consistent across the team.
- `direction/steer` — Set design direction, surface tradeoffs, and guide portfolio decisions so effort lands where it matters most.

## operations/

- `operations/govern` — Set the standards, decision rights, review paths, and exception handling that keep design operations consistent and defensible over time.
- `operations/instrument` — Define, audit, and verify the metrics and signals that let design operations measure process health and service quality.
- `operations/schedule` — Plan recurring cadences, milestones, and capacity so design operations work happens at the right time and does not overload the team.
- `operations/systematize` — Build repeatable design operations processes, templates, and handoffs so the team can run work with less ambiguity and less manual coordination.

# Normalize

Use this skill to realign a feature or component set that has drifted away from system standards.

## When to Use

- When a feature uses inconsistent spacing, tokens, variants, or component patterns
- When local customizations have weakened design-system coherence
- When a surface needs to be brought back in line before broader rollout or reuse

## How to Use

Compare the target feature against the active system across components, tokens, naming, states, theming, and composition patterns. Separate true system gaps from unnecessary drift. Replace ad hoc solutions with existing system assets wherever the fit is legitimate.

If the feature exposes a real missing capability, frame it as a system extension proposal instead of hiding it behind a local exception. Normalization should leave the surface more consistent and the system more honest.

## What to Produce

- A gap analysis showing where the feature drifted from the system
- A normalized mapping to approved components, tokens, and patterns
- Any documented system gaps that require formal extension rather than local workaround

## Notes for Design Systems Designer

Do not force-fit the feature into the wrong system asset just to make it look compliant. Good normalization strengthens both the product and the system.
